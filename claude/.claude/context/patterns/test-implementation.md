# テスト実装の3原則

テストコードは以下の3つの観点で評価される。実装時にこの原則を意識することで、高品質なテストを作成できる。

## 1. 無駄なテストがないか

### 避けるべきパターン
- **過剰なモック**: 実装の詳細に依存しすぎるモック（実装変更でテストが壊れる）
- **重複テスト**: 同じ振る舞いを複数のテストでカバー
- **意味のないテスト**: 実際には何も検証していないテスト

### 良い例
- 必要最小限のモック
- 各テストが独自の価値を持つ
- 実装の変更に強いテスト

## 2. 必要なテストの抜け漏れがないか

### 確認すべき項目
- **正常系**: 期待される動作のテスト
- **異常系**: エラーケース、例外処理のテスト
- **エッジケース**: 境界値、NULL、空文字列、最大値等

### 良い例
- 主要な実行パスがすべてカバーされている
- エラーハンドリングが適切にテストされている
- ビジネスロジックの重要な分岐が網羅されている

## 3. テストの可読性

### 重要な要素
- **テスト名**: 何をテストしているか明確（例: `should return error when input is null`）
- **構造**: Arrange-Act-Assert パターン等の一貫した構造
- **意図の明確さ**: なぜこのテストが必要かが分かる

### 良い例
- テストケースを読めば仕様が理解できる
- テスト失敗時に原因が特定しやすい
- 将来のメンテナが理解しやすい

## 適用場面

- テストコード実装時の指針として
- コードレビュー時の評価基準として
- リファクタリング時の品質チェックとして
