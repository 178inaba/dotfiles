# finding-triage

レビュー指摘・修正提案（finding）を「対応しない」と確定する前の検証規律。見送り判断は独立レビューの検証網に掛からない唯一の箇所で、印象ベースの見積もりが素通りしやすい（「7ファイル以上に波及」を理由に見送った修正が、実際は機械的追従9ファイルで根本修正できた事例あり）。本スキルはその箇所に検証を強制する。

## 適用対象

- **対応が期待される指摘**（blocking 相当の指摘・改善提案）の見送り判断に適用する
- 質問・確認事項（回答すべきもの・ユーザーの判断に残すもの）の扱いは各消費者スキル側の責務であり、本スキルの対象外
- **消費者の写像規定**: 本スキルを参照するスキルは、自分の重要度ラベルから「対応が期待される指摘」への対応を自スキル内に1行で定める（例: deep-review の「必須修正・推奨修正」、Conventional Comments の blocking / non-blocking / nit、/simplify の finding）

## 規律

### 1. 見送り理由の実地検証

見送り理由に含まれる事実主張（影響範囲・前提コードの挙動・仕様/Issue の記述）は、印象・記憶で確定せず、grep・対象コードの再読・原文の再確認で検証してから確定する。

### 2. コスト理由の見送りは波及の分類から

コスト理由（「波及が大きい」等）の見送りは、呼び出し元・影響先を実際に列挙し、変更が機械的追従で済むか判断を要するかを分類してから判断する。機械的追従なら波及ファイル数が多くても低コストであり、見送り理由にならない。

### 3. 非必須ルールの一様準拠判定

「推奨」等の非必須と記載されたルールからの逸脱（またはそのルールを根拠にした見送り）は、変更対象ファイル（同種の適用箇所が無い場合は同レイヤーの周辺コード）の既存コードが一様に従っているかを、近傍の同種適用箇所数件で確認してから確定する。一様に従っている場合は事実上の必須慣習として扱う — 「非必須だから違反してよい」は見送り根拠にならない。

### 4. 検証根拠の記録

見送りには検証根拠（実行した grep・再読した箇所・確認した原文の記述）を添付する。書けない場合は検証未実施のサインなので、判断に戻って検証してから確定する。

### 5. 繰り返される同種見送りの規約昇格

同種の指摘を同じ理由で繰り返し見送っている場合、個別の見送りを続けず、見送り理由を規約側へ昇格して再指摘を構造的に止める。昇格先は「コンテキスト管理」の分離基準（トリガーの性質）で選ぶ — 常時効くべき方針はプロジェクト/個人 CLAUDE.md、特定ファイル起点なら rules/、レビュー観点そのものの調整ならレビュー系スキルの scope 規定。理由: 妥当な見送りでも反復されるとレビュー出力全体の信頼を侵食し、棄却が積み重なるとレビュー自体が読まれなくなる（false positive 運用の一般知見）。
