# /issue-create

壁打ちで固めた内容を「迷わず実装できる」GitHub Issue として構造化・作成する。

## 使用方法
```
/issue-create              # 直前の会話から要件を抽出
/issue-create <トピック>   # 会話のうち指定トピックに絞って抽出
```

## 前提条件
- Git リポジトリ内で実行すること（対象リポジトリの判定に使用）
- `gh` CLI がインストール・認証済みであること

## 実行内容

### 1. 対象リポジトリ確定
- `gh repo view --json nameWithOwner -q .nameWithOwner` で取得し、以降の書き込み系で `-R` に渡す

### 2. 素材の蒸留
- 会話コンテキスト（引数があればそのトピックに絞る）から要件を抽出する
- **「会話の要約」ではなく「要件の蒸留」**: 経緯・試行錯誤は捨て、合意した結論のみを残す
- 合意事項を**番号付きリスト**として書き出す（手順 7 の突き合わせに使う）。以下を漏らさず含める:
  - やらないと決めたこと（スコープ外）
  - 採用しなかった案と、採用しなかった理由（設計判断の再検討を防ぐ）
  - 意図的に保留した判断（TBD）と、誰が・いつ決めるか
- セルフチェック: 会話で暗黙に合意したが明文化していない判断の拾い漏れがないか確認する

### 3. 重複検索
- `gh issue list --search "<キーワード>" --state all` で類似 Issue を確認（実装済み・wontfix の closed も重複検出の対象）
- 類似があればユーザーに提示し、続行可否を確認する

### 4. 粒度判定
- 基準: **1 つのレビュー可能な PR で完結するか**。実装を伴う Issue は予想 diff 行数で定量判定する:
  - 理想 200〜400 行 / 許容 〜800 行 / **800 行超は分割**（テストコード込み。一般的なコードレビューのベストプラクティス基準）
  - 見積りは対象リポジトリの類似実装（同種のテーブル追加・API 追加等の実ファイル群）の行数から概算する
  - 理由: 実装途中で「PR が大きすぎる」と判明してからの分割は手戻りが大きい。起票時に閾値以下へ分割しきる
- 収まる → 単一 Issue として進行
- 超える → 縦切り（機能として完結する単位。層別のフロント/バック分割は避ける）の分割案を提示し、承認後に親 Issue + Sub-Issues 構成へ
  - 縦切りしても閾値を超える単位は、依存順の段階分割（schema → Repository → API・バッチ等）まで下げてよい
  - 分割後の**各 Sub にも同じ定量基準を適用**し、見積り行数と依存関係（並列実装の可否）を添えて提示する

### 5. ギャップ調査
- 本文テンプレートの各節を埋める観点で、会話に無い・未確定の情報を列挙し（壁打ちが薄い場合も全節を確認）、事実か判断かで扱いを分ける:
  - **事実のギャップ** → コードベース・CI ログ・`gh` 等の追加調査で埋める（独立した調査は並列実行する）。特に:
    - **原因・現状認識**: 仮説のままにせず、コードや記録を読んで裏付ける
    - **参照コード**: 実装者が読みに行く前提にせず、短い該当箇所は本文に引用する
    - **再現・検証手順**: 実際に機能することを確認する
  - **判断のギャップ**（要件の取捨・スコープ・受け入れ水準等）→ 調査では埋まらないため、ドラフト前にユーザーに確認する
- 合理的な調査で埋まらない事実は、隠さず「未確認」と明示し、確認方法とともに本文に残す（判断は「未確認」のまま実装者に渡さない）
- テンプレートに対応節が無い情報（再現手順等）は最も近い節に含める
- 調査結果が重複・粒度の前提を変えた場合は手順 3〜4 を再評価する

### 6. 言語・ラベル判定
以下の2コマンドは独立のため並列実行する:
- `gh issue list --limit 5` で既存 Issue の言語慣例（日本語/英語）を確認し、それに従う
- `gh label list` で既存ラベルを取得し、内容に合うものを選定（合うものが無ければ付けない。新規ラベルは作らない）

### 7. 自己完結検証（ドラフト提示前に必須）

ドラフトは会話に直接書かず、**最初から scratchpad のファイルとして Write する**（分割時は親・各 Sub-Issue で 1 ファイルずつ）。以降の修正はすべてこのファイルへの Edit で行い、手順 9 では同じファイルを `--body-file` に渡す。理由: 承認済み文面を作成時に再転記することによるドリフトと、長い会話のコンパクションによる細部欠落を防ぐ。

ドラフト作成者自身のチェックは「会話を知っているから読めてしまう」バイアスで穴を見逃すため、以下の 2 方向の検証を必ず行う:

1. **合意網羅チェック（会話 → 本文）**: 手順 2 の合意事項リストの全項目を、ドラフト本文と 1 項目ずつ突き合わせる。本文に無い項目は追記する（分割時はどの Sub-Issue が担うかも確認）
2. **新規読者チェック（本文 → 読者）**: 会話コンテキストを持たないサブエージェント（Explore 等の読み取り専用）に**ドラフトファイルのパスのみ**を渡して Read させ、次を依頼する:
   > あなたはこの Issue にアサインされた実装者で、起案時の議論は一切読めない。この Issue 単体で実装に着手できるか評価し、(a) 調査の起点・方針の選択・検証方法で迷う点、(b) 本文外の前提知識を要求している箇所、(c) 複数解釈できる記述、を列挙せよ。問題なければ「指摘なし」と返せ。

指摘への対処は性質で分ける:
- **事実の不足** → 手順 5 のギャップ調査で埋めて本文に追記
- **判断の不足** → ユーザーに確認（勝手に決めない）
- **実装者の裁量に委ねてよい事項** → 「実装時判断に委ねる事項」節に推奨・判断基準つきで明記（見落としと意図的な保留を読者が区別できるようにする）

修正したら新規読者チェックを**新しいサブエージェントで再実行**し、「指摘なし」になるまで繰り返す（通常 1〜2 周で収束。3 周しても新規指摘が続く場合は残指摘をユーザーに提示して判断を仰ぐ）。

過剰詳細化ガード: 対象リポジトリの既存慣習・参照ドキュメントから一意に決まる内容は追記しない（本文が長いほど実装成功率が下がる。「〜の既存パターンに従う」の一文と参照先で足りる）。

### 8. ドラフト提示 → ユーザー承認
- タイトル・本文・ラベル（分割時は親子構成と親・各 Sub-Issue の本文すべて）を提示し、承認を得てから作成する
- 承認対象は**ファイルの最終版全文**とする（修正を重ねた場合、差分の積み重ねだけで承認させない）
- Issue は外向き公開物のため、承認は省略しない
- ユーザー修正はファイルへの Edit で反映する。修正が要件の追加・変更を含むなら手順 7 を再実行してから再提示する（表現の微修正のみなら不要）

### 9. 作成
- 手順 7〜8 で維持したドラフトファイルをそのまま指定: `gh issue create -R <repo> --title "..." --body-file <path> --assignee @me [--label ...]`（承認後に本文を書き直さない）
- 分割時: 親 Issue を作成後、Sub-Issue ファイル内の親番号等のプレースホルダーを Edit で置換し、@~/.claude/skills/github-sub-issues/SKILL.md の手順で Sub-Issues を作成・リンク

### 10. 完了報告
- Issue URL を提示し、次アクションとして `/issue-handle <番号>` を案内する

## 本文テンプレート

```markdown
## 背景・目的
（なぜこの作業が必要か。担当者が判断に迷ったとき立ち返る基準）

## 要件
（must の列挙。曖昧語は具体化する — 「速くする」ではなく「0.5秒以内」）

## 受け入れ条件
- [ ] （観測可能な振る舞いで書く。実装詳細は書かない）
- [ ] （エッジケースを明記する）

## 影響範囲・関連コード
（ファイルパス・grep 可能な識別子・関連 Issue/PR リンク）

## 実装方針の示唆（任意）
（断定せず「案」として。複数案あるなら推奨案とその理由を明示する。検討して採用しなかった案は不採用理由も書く — 実装者による再検討・逆行を防ぐ。不要なら節ごと削除）

## 実装時判断に委ねる事項（任意）
（意図的にオープンにする判断の列挙。各項目に推奨と判断基準を添える。見落としではなく意図的な保留であることを読者に示す。不要なら節ごと削除）

## スコープ外
（壁打ちで「やらない」と決めたことの明記）
```

## 品質基準

ドラフト提示前に以下を自己チェックする（共通原則: 未解決の判断や裏付けの無い主張をそのまま実装者に渡さない）:
- [ ] タイトル: 命令形の具体的な一文（後から見て文脈がわかる）
- [ ] 自己完結: 壁打ちの会話を読まなくても Issue 単体で実装可能 — 手順 7 の 2 方向検証（合意網羅チェック・新規読者チェック）を完了し、「指摘なし」に到達している
- [ ] 粒度: 実装を伴う Issue は予想 diff が 800 行以下（分割時は各 Sub とも）— 手順 4 の定量判定を通過している
- [ ] concise but complete: 冗長化させない（本文が長いほど AI エージェントの実装成功率が下がる）
- [ ] 受け入れ条件の完全性: 全項目が満たされたら、かつその時のみ完了と言える
- [ ] 事実と仮説の区別: 裏付けの無い記述を検証済みであるかのように書いていない
