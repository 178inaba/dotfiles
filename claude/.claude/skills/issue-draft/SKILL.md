# /issue-draft

壁打ちで固めた内容を「迷わず実装できる」GitHub Issue として構造化・作成する。`--refine` では既存 Issue を素材に、同じ品質基準へブラッシュアップして本文を更新する。

## 使用方法
```
/issue-draft                   # 直前の会話から要件を抽出して新規作成
/issue-draft <トピック>        # 会話のうち指定トピックに絞って抽出
/issue-draft --refine <番号>   # 既存 Issue をブラッシュアップして本文を更新
```

`--` で始まる引数が `--refine <番号>` 形式でない場合は、トピックとして解釈せずエラーで停止する（typo が意図しない新規作成に落ちるのを防ぐ）。

以降の手順は新規作成を基本フローとし、`--refine` 時の差分のみ各手順内に「`--refine` 時」として示す。差分が適用されるのは refine の対象 Issue のみで、分割時に新規作成する Sub-Issues には新規作成のルール（`--assignee @me`・ラベル選定）をそのまま適用する。

## 前提条件
- Git リポジトリ内で実行すること（対象リポジトリの判定に使用）
- `gh` CLI がインストール・認証済みであること

## 実行内容

### 1. 対象リポジトリ確定
- `gh repo view --json nameWithOwner -q .nameWithOwner` で取得し、以降の書き込み系で `-R` に渡す

### 2. 素材の蒸留
- 会話コンテキスト（トピック引数があればそのトピックに絞る）から要件を抽出する
- **`--refine` 時**: 素材は既存 Issue の本文とコメント欄の議論。`gh issue view <番号> --json title,body,labels,assignees,author,comments` で一括取得し、後続手順（言語・ラベル判定、変更点の提示、編集通知の要否判断）でもこのスナップショットを再利用する（コンパクション等でコンテキストから欠落した場合のみ再取得する）。会話コンテキストがあれば補完に使う
- **「会話の要約」ではなく「要件の蒸留」**: 経緯・試行錯誤は捨て、合意した結論のみを残す
- 合意事項を**番号付きリスト**として書き出す（手順 7 の突き合わせに使う）。以下を漏らさず含める:
  - やらないと決めたこと（スコープ外）
  - 採用しなかった案と、採用しなかった理由（設計判断の再検討を防ぐ）
  - 意図的に保留した判断（TBD）と、誰が・いつ決めるか
- セルフチェック: 会話で暗黙に合意したが明文化していない判断の拾い漏れがないか確認する

### 3. 重複検索
- `gh issue list --search "<キーワード>" --state all` で類似 Issue を確認（実装済み・wontfix の closed も重複検出の対象）
- 類似があればユーザーに提示し、続行可否を確認する
- **`--refine` 時**: 対象 Issue 自身は検索結果から除外する（ブラッシュアップ対象が別の Issue と重複している可能性は新規作成時と同様にある）

### 4. 粒度判定
- 基準: **リリース単位の atomicity（greenfield 理想）**。機能として価値を出すために一緒にリリースすべき範囲は、予想 diff が大きくても単一 Issue とする
- 分割は greenfield でも自然な境界がある場合のみ提案し、承認後に親 Issue + Sub-Issues 構成へ:
  - 独立してリリース可能で、単体で価値が完結する縦切り。層別のフロント/バック分割や依存順の段階分割（schema → Repository → API 等）は境界として認めない — atomic な機能を工程で割った断片で、単体では価値が完結しないため
  - 時間的に必然分離される作業（デプロイ完了後の互換コード除去等、PR が別になることが確定しているもの）
  - 分割後の**各 Sub にも同じ atomicity 基準を適用**し、依存関係（並列実装の可否）を添えて提示する
  - **`--refine` 時**: 対象 Issue を親とし、その本文を親用に書き直した上で Sub-Issues を新規作成する
- プロセス都合（レビューサイズ等）の分割は、ユーザーが明示的に指示した場合に限り行う。その際も greenfield 理想からの妥協であることと妥協で失われる特性（リリースの atomicity 等）を明示する（設計判断原則「妥協は妥協と明示する」に従う）
- 分割とは別に、**価値の取捨チェック**を行う: Issue 内に「削っても残りの機能が成立する、相対的に価値の薄い部分」が無いか確認する。見つかった場合はスコープ外へ移す・後続 Issue に切り出す選択肢を理由とともに提示し、ユーザーの判断を仰ぐ（勝手に削らない — 仕様遵守原則）。手順 7 のデッドスペックチェックが検出するのは「消費者の無い構成要素」であり、消費者はいるが価値の薄い機能はこのチェックでしか拾えない（サイズ都合の分割を排した本基準では、分割の過程が副産物として提供する「低価値部分の発見と破棄」の機会が失われるため、その機会をここで直接確保する）

### 5. ギャップ調査
- 本文テンプレートの各節を埋める観点で、会話に無い・未確定の情報を列挙し（壁打ちが薄い場合も全節を確認）、事実か判断かで扱いを分ける:
  - **事実のギャップ** → コードベース・CI ログ・`gh` 等の追加調査で埋める（独立した調査は並列実行する）。特に:
    - **原因・現状認識**: 仮説のままにせず、コードや記録を読んで裏付ける
    - **参照コード**: 実装者が読みに行く前提にせず、短い該当箇所は本文に引用する。参照はファイルパス + 関数名・型名等の grep 可能な識別子で示し、**行番号は書かない**（起票から実装までにズレて誤参照になる上、手順 7 の検証で行番号の突き合わせ・軽微なズレ指摘にコストを浪費するため）
    - **再現・検証手順**: 実際に機能することを確認する
  - **判断のギャップ**（要件の取捨・スコープ・受け入れ水準等）→ 調査では埋まらないため、ドラフト前にユーザーに確認する
- 合理的な調査で埋まらない事実は、隠さず「未確認」と明示し、確認方法とともに本文に残す（判断は「未確認」のまま実装者に渡さない）
- テンプレートに対応節が無い情報（再現手順等）は最も近い節に含める
- 調査結果が重複・粒度の前提を変えた場合は手順 3〜4 を再評価する

### 6. 言語・ラベル判定
以下の2コマンドは独立のため並列実行する:
- `gh issue list --limit 5` で既存 Issue の言語慣例（日本語/英語）を確認し、それに従う
- `gh label list` で既存ラベルを取得し、内容に合うものを選定（合うものが無ければ付けない。新規ラベルは作らない）
- **`--refine` 時**: 言語確認（`gh issue list`）はスキップし、言語は対象 Issue 本文に従う（分割時の Sub-Issues も同様）。`gh label list` のみ実行し、既存ラベルは維持した上で内容に合う追加候補があればドラフト提示時に提案する

### 7. 自己完結検証（ドラフト提示前に必須）

ドラフトは会話に直接書かず、**最初から scratchpad のファイルとして Write する**（分割時は親・各 Sub-Issue で 1 ファイルずつ）。以降の修正はすべてこのファイルへの Edit で行い、手順 9 では同じファイルを `--body-file` に渡す。理由: 承認済み文面を作成時に再転記することによるドリフトと、長い会話のコンパクションによる細部欠落を防ぐ。

ドラフト作成者自身のチェックは「会話を知っているから読めてしまう」バイアスで穴を見逃すため、作成者自身のチェック（1・2）に加えて、会話コンテキストを持たない読者による検証（3）を必ず行う:

1. **合意網羅チェック（会話 → 本文）**: 手順 2 の合意事項リストの全項目を、ドラフト本文と 1 項目ずつ突き合わせる。本文に無い項目は追記する（分割時はどの Sub-Issue が担うかも確認）
2. **デッドスペックチェック（構成要素 → 消費者）**: 本文が指定する具体的な構成要素（構造体フィールド・環境変数・引数・設定値・ファイル等）それぞれについて、それを消費する要件・受け入れ条件を確認する。消費者が無い要素は削除するか、維持する理由と消費者（後続 Issue・外部システム等）を本文に明記する。参考実装からのパターン踏襲で紛れ込みやすく、「既存との一貫性」は消費者の代わりにならない（実装者は仕様遵守原則に従い本文の要素を勝手に削除できないため、消費者の無い要素はデッドコードとしてそのまま実装され、レビュー指摘に対しても「Issue に明記」を根拠に防衛された事例あり）
3. **新規読者チェック（本文 → 読者）**: 会話コンテキストを持たないサブエージェント（`subagent_type: "fresh-reader"`。model はエージェント定義で opus に固定、tools も読み取り系に固定。呼び出し時に `model` は指定しない — 指定漏れで親モデルを継承する事故を構造的に防ぐ。opus を充てる理由: チェックの主成分（自己完結性・複数解釈の検出）は新規読者性に依存しモデルティアで差が出にくく、Issue の欠陥は下流の issue-handle 側の検問（要件対話・計画検証・deep-review）でも捕捉されるため、コスト原則「差が出ない領域は下位ティア」に従い fable は充てない）に**ドラフトファイルのパスのみ**を渡して Read させ、次を依頼する:
   > あなたはこの Issue にアサインされた実装者で、起案時の議論は一切読めない。この Issue 単体で実装に着手できるか評価し、指摘を次の 2 分類で列挙せよ:
   > - **blocker**: 事実誤認（存在しない関数・ファイル・実物と異なる参照や引用）／間違った実装を招く複数解釈／実装に着手できないレベルの情報欠落
   > - **minor**: 参照先のコード・ドキュメントを読めば解決できる詳細／実装者の合理的裁量で決められる事項／「あると良い」レベルの改善提案
   >
   > 分類に迷う指摘は minor に倒せ。blocker が無ければ、minor の有無にかかわらず冒頭で「blocker なし」と明言せよ。

指摘への対処は分類で分ける:
- **blocker（事実の不足）** → 手順 5 のギャップ調査で埋めて本文に追記
- **blocker（判断の不足）** → ユーザーに確認（勝手に決めない）
- **minor** → 原則本文に反映しない（過剰詳細化ガードと同じ理由）。実装者が「見落とし」と誤解しそうな項目のみ、「実装時判断に委ねる事項」節に推奨・判断基準つきの 1 行で吸収する

収束条件は「blocker なし」（minor が残っていても収束とする）。1 周目で blocker が出た場合のみ、反映後に 2 周目を実行する。2 周目は新しいサブエージェントを起動せず、`SendMessage` で 1 周目のサブエージェントに差し戻す（新規読者性が必要なのは初読の 1 周目のみ。2 周目の職務は「指摘した blocker が修正で解消されたか」の検証であり、自分の指摘と読了済みドラフトのコンテキストを保持した同一レビュアーが適任。コンテキスト維持により再読込がキャッシュ読取になり、コストも抑えられる）。依頼文には次を含め、職務を絞る:
   > 本文へ次の修正を反映した: <blocker への修正点の要約>。ドラフトファイルを再度 Read し、検証対象は (a) あなたが指摘した blocker がこの修正で解消され、本文全体と矛盾なく反映されているか、(b) 修正が新たに生んだ blocker、のみ。それ以外の箇所への新規指摘は blocker 基準を満たす場合に限り報告せよ。コードベースの網羅的な再検証は行わない。分類基準と「迷ったら minor」は初回と同じ。blocker が無ければ冒頭で「blocker なし」と明言せよ。

`SendMessage` で 1 周目のエージェントに到達できない場合（エージェント終了・セッション断絶等）のみ、フォールバックとして会話コンテキストを持たない新しいサブエージェント（初回と同じ起動条件）を起動し、初回の役割・分類基準と上記の職務限定を渡して 2 周目を実行する。

検証は**最大 2 周**。2 周目でも blocker が残る場合は、反映後に**3 周目へ自律的に入らず**、未収束である旨・残る blocker の内容・続行/打ち切りの選択肢をユーザーに提示して判断を仰ぐ（続行を選んだ場合の 3 周目以降も同一エージェントへの `SendMessage` 差し戻しで行う）。

過剰詳細化ガード: 対象リポジトリの既存慣習・参照ドキュメントから一意に決まる内容は追記しない（本文が長いほど実装成功率が下がる。「〜の既存パターンに従う」の一文と参照先で足りる）。

### 8. ドラフト提示 → ユーザー承認
- タイトル・本文・ラベル（分割時は親子構成と親・各 Sub-Issue の本文すべて）を提示し、承認を得てから作成・更新する
- 承認対象は**ファイルの最終版全文**とする（修正を重ねた場合、差分の積み重ねだけで承認させない）
- **`--refine` 時**: 全文に加えて現行本文からの主な変更点を添えて提示する
- Issue は外向き公開物のため、承認は省略しない
- ユーザー修正はファイルへの Edit で反映する。修正が要件の追加・変更を含むなら手順 7 を再実行してから再提示する（表現の微修正のみなら不要）

### 9. 作成・更新
- 手順 7〜8 で維持したドラフトファイルをそのまま指定する（承認後に本文を書き直さない）
- 新規作成: `gh issue create -R <repo> --title "..." --body-file <path> --assignee @me [--label ...]`
- **`--refine` 時**: `gh issue edit <番号> -R <repo> --body-file <path> [--title "..."] [--add-label ...]`。アサインは変更しない
  - 起案者・コメント参加者に自分以外がいる場合（自分のログイン名は `gh api user -q .login` で確認）は、手順 8 で提示した変更点を本文とする編集通知コメントを提案し、承認を得て投稿する（ファイルに書き出して `--body-file` で渡す。既読者が黙った書き換えに気付けないままになるのを防ぐ）
- 分割時: 親 Issue を作成（`--refine` 時は更新）後、Sub-Issue ファイル内の親番号等のプレースホルダーを Edit で置換し、@~/.claude/skills/github-sub-issues/SKILL.md の手順で Sub-Issues を作成・リンク

### 10. 完了報告
- Issue URL を提示し、次アクションとして `/issue-handle <番号>` を案内する

## 本文テンプレート

```markdown
## 背景・目的
（なぜこの作業が必要か。担当者が判断に迷ったとき立ち返る基準）

## 要件
（must の列挙。曖昧語は具体化する — 「速くする」ではなく「0.5秒以内」）

## 受け入れ条件
- [ ] （観測可能な振る舞いで書く。実装詳細は書かない）
- [ ] （エッジケースを明記する）

## 影響範囲・関連コード
（ファイルパス・grep 可能な識別子（関数名・型名等。行番号は書かない）・関連 Issue/PR リンク）

## 実装方針の示唆（任意）
（断定せず「案」として。複数案あるなら推奨案とその理由を明示する。検討して採用しなかった案は不採用理由も書く — 実装者による再検討・逆行を防ぐ。不要なら節ごと削除）

## 実装時判断に委ねる事項（任意）
（意図的にオープンにする判断の列挙。各項目に推奨と判断基準を添える。見落としではなく意図的な保留であることを読者に示す。不要なら節ごと削除）

## スコープ外
（壁打ちで「やらない」と決めたことの明記）
```

## 品質基準

ドラフト提示前に以下を自己チェックする（共通原則: 未解決の判断や裏付けの無い主張をそのまま実装者に渡さない）:
- [ ] タイトル: 命令形の具体的な一文（後から見て文脈がわかる）
- [ ] 自己完結: 壁打ちの会話を読まなくても Issue 単体で実装可能 — 手順 7 の検証（合意網羅チェック・デッドスペックチェック・新規読者チェック）を完了し、「blocker なし」に到達している
- [ ] 粒度: 分割がリリース単位の atomicity 基準を満たしている（プロセス都合の分割はユーザーが明示的に指示した妥協のみ）— 手順 4 の判定を通過している
- [ ] concise but complete: 冗長化させない（本文が長いほど AI エージェントの実装成功率が下がる）
- [ ] 受け入れ条件の完全性: 全項目が満たされたら、かつその時のみ完了と言える
- [ ] 事実と仮説の区別: 裏付けの無い記述を検証済みであるかのように書いていない
