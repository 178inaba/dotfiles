# /deep-plan-review

実装計画を、会話コンテキストを持たない読み取り専用サブエージェントに検証させ、blocker が無くなるまで計画を修正する。計画起案者自身のチェックは「議論を知っているから読めてしまう」バイアスで穴を見逃すため、新規読者による検証を挟む。

## 使用方法
```
/deep-plan-review                     # 現在の plan モードの計画ファイルを検証
/deep-plan-review path/to/plan.md     # 計画ファイルを明示指定
```

## 実行内容

### 1. 対象計画の特定
- 引数でファイルパスが指定されていればそれを使用
- 省略時は現在の plan モードの計画ファイル（plan モード開始時に指定されたパス）
- どちらも無い場合はエラーで停止し、計画ファイルの指定を促す

### 2. 検証サブエージェントの起動
- Agent ツールで読み取り専用サブエージェント（Explore 等）を `model: "fable"` で起動する（計画の検証は対象が計画とその参照範囲に限定され、判断密度が高い批評タスクのため、上位モデルを充てる）
- **計画ファイルのパスのみ**を渡す（会話コンテキストを渡さない = 新規読者性の担保）
- 依頼文に以下を含める:
  > あなたはこの計画で実装を任された実装者で、計画起案時の議論は一切読めない。計画ファイルを Read し、計画単体で実装に着手できるかを以下の観点で評価せよ。計画が参照するファイル・関数・設定・コマンドは Read/Grep で実在と内容を実地検証すること。
  > - 参照実在性: 存在しないファイル・関数・設定・コマンドを参照していないか
  > - 前提と実コードの一致: 計画が前提とする既存コードの構造・挙動が実物と一致しているか
  > - 設計妥当性: 手順が計画の目的・要件と整合しているか、見落としているリスクはないか
  > - 自己完結性: この計画単体で迷わず実装可能か
  >
  > 指摘を次の 2 分類で列挙せよ:
  > - **blocker**: 事実誤認（存在しない関数・ファイル・実物と異なる参照や引用）／間違った実装を招く複数解釈／実装に着手できないレベルの情報欠落
  > - **minor**: 参照先のコード・ドキュメントを読めば解決できる詳細／実装者の合理的裁量で決められる事項／「あると良い」レベルの改善提案
  >
  > 分類に迷う指摘は minor に倒せ。blocker が無ければ、minor の有無にかかわらず冒頭で「blocker なし」と明言せよ。

### 3. 指摘の計画反映
- **blocker（事実の不足・誤り）** → 調査して計画を修正
- **blocker（判断の不足）** → ユーザーに確認（勝手に決めない）
- **minor** → 原則反映しない（計画の過剰詳細化を防ぐ。実装中の裁量に委ねる）

### 4. blocker 収束
- 収束条件は「blocker なし」（minor が残っていても収束とする）
- 1 周目で blocker が出た場合のみ、反映後に 2 周目を実行する。2 周目も会話コンテキストを持たない**新しい**サブエージェント（手順 2 と同じ起動条件）に計画全文を Read させるが、依頼文には手順 2 の役割・分類基準（検証観点と「全参照の実地検証」指示は含めない）に加えて次を含め、職務を絞る:
  > 直前に計画へ次の修正を反映した: <blocker への修正点の要約>。検証対象は (a) この修正が計画全体と矛盾なく反映されているか、(b) 修正が新たに生んだ blocker、のみ。Read/Grep による実地検証は修正が触れたファイル・関数等の参照に限る。それ以外の箇所への新規指摘は blocker 基準を満たす場合に限り報告せよ。コードベースの網羅的な再検証は行わない。
- 検証は**最大 2 周**。2 周目でも blocker が残る場合は、反映後に 3 周目へ自律的に入らず、未収束である旨・残る blocker の内容・続行/打ち切りの選択肢をユーザーに提示して判断を仰ぐ

### 5. plan モード復帰
- plan モード中の場合のみ、修正後の計画で `ExitPlanMode` を呼び出す
- plan モード外で起動された場合は、レビュー結果と反映内容の報告のみで終了

## 注意事項
- @~/.claude/skills/check-plan-compliance/SKILL.md（プロジェクト規約への準拠検証）と併用する場合は **compliance 先 → 本スキル後** の順で実行する（高コストな検証は最終形の計画に 1 回だけ充てるため）。その場合 ExitPlanMode は本スキル側が担い、compliance 側では呼ばない
- `fable` は 2026-07-20 以降サブスクリプション外の従量課金。モデル方針を変更する場合は手順 2 の指定を編集する
- `fable` が利用不可（availableModels 許可リスト外）の環境では、Claude Code が警告の上で親モデル継承にフォールバックするため実行自体は継続する（v2.1.207 のコード解析で確認、2026-07-14）
- 検証サブエージェントは読み取り専用で、計画の修正は本スキル（親セッション）が行う
