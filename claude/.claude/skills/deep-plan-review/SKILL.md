# /deep-plan-review

実装計画を、会話コンテキストを持たない読み取り専用サブエージェントに検証させ、blocker が無くなるまで計画を修正する。計画起案者自身のチェックは「議論を知っているから読めてしまう」バイアスで穴を見逃すため、新規読者による検証を挟む。

## 使用方法
```
/deep-plan-review                     # 現在の plan モードの計画ファイルを検証
/deep-plan-review path/to/plan.md     # 計画ファイルを明示指定
```

## 実行内容

### 1. 対象計画の特定
- 引数でファイルパスが指定されていればそれを使用
- 省略時は現在の plan モードの計画ファイル（plan モード開始時に指定されたパス）
- どちらも無い場合はエラーで停止し、計画ファイルの指定を促す

### 2. 検証サブエージェントの起動
- Agent ツールで読み取り専用サブエージェント（Explore 等）を `model: "opus"` で起動する（検証の主成分 — 参照実在性・前提と実コードの一致・自己完結性 — は Read/Grep の実地検証と新規読者性に依存し、モデルティアで差が出にくいため、コスト原則「差が出ない領域は下位ティア」に従い fable は充てない）
- **計画ファイルのパスのみ**を渡す（会話コンテキストを渡さない = 新規読者性の担保）
- 依頼文に以下を含める:
  > あなたはこの計画で実装を任された実装者で、計画起案時の議論は一切読めない。計画ファイルを Read し、計画単体で実装に着手できるかを以下の観点で評価せよ。計画が参照するファイル・関数・設定・コマンドは Read/Grep で実在と内容を実地検証すること。
  > - 参照実在性: 存在しないファイル・関数・設定・コマンドを参照していないか
  > - 前提と実コードの一致: 計画が前提とする既存コードの構造・挙動が実物と一致しているか
  > - 設計妥当性: 手順が計画の目的・要件と整合しているか、見落としているリスクはないか
  > - 自己完結性: この計画単体で迷わず実装可能か
  >
  > 指摘を次の 2 分類で列挙せよ:
  > - **blocker**: 事実誤認（存在しない関数・ファイル・実物と異なる参照や引用）／間違った実装を招く複数解釈／実装に着手できないレベルの情報欠落
  > - **minor**: 参照先のコード・ドキュメントを読めば解決できる詳細／実装者の合理的裁量で決められる事項／「あると良い」レベルの改善提案
  >
  > 分類に迷う指摘は minor に倒せ。blocker が無ければ、minor の有無にかかわらず冒頭で「blocker なし」と明言せよ。

### 3. 指摘の計画反映
- **blocker（事実の不足・誤り）** → 調査して計画を修正
- **blocker（判断の不足）** → ユーザーに確認（勝手に決めない）
- **minor** → 原則反映しない（計画の過剰詳細化を防ぐ。実装中の裁量に委ねる）

### 4. blocker 収束
- 収束条件は「blocker なし」（minor が残っていても収束とする）
- 1 周目で blocker が出た場合のみ、反映後に 2 周目を実行する。2 周目は新しいサブエージェントを起動せず、`SendMessage` で 1 周目のサブエージェントに差し戻す（新規読者性が必要なのは初読の 1 周目のみ。2 周目の職務は「指摘した blocker が修正で解消されたか」の検証であり、自分の指摘と読了済みファイルのコンテキストを保持した同一レビュアーが適任。コンテキスト維持により計画・参照ファイルの再読込がキャッシュ読取になり、コストも抑えられる）。依頼文には次を含め、職務を絞る:
  > 計画へ次の修正を反映した: <blocker への修正点の要約>。計画ファイルを再度 Read し、検証対象は (a) あなたが指摘した blocker がこの修正で解消され、計画全体と矛盾なく反映されているか、(b) 修正が新たに生んだ blocker、のみ。Read/Grep による実地検証は修正が触れたファイル・関数等の参照に限る。それ以外の箇所への新規指摘は blocker 基準を満たす場合に限り報告せよ。コードベースの網羅的な再検証は行わない。分類基準と「迷ったら minor」は初回と同じ。blocker が無ければ冒頭で「blocker なし」と明言せよ。
- `SendMessage` で 1 周目のエージェントに到達できない場合（エージェント終了・セッション断絶等）のみ、フォールバックとして会話コンテキストを持たない新しいサブエージェント（手順 2 と同じ起動条件）を起動し、手順 2 の役割・分類基準（検証観点と「全参照の実地検証」指示は含めない）と上記の職務限定を渡して 2 周目を実行する
- 検証は**最大 2 周**。2 周目でも blocker が残る場合は、反映後に 3 周目へ自律的に入らず、未収束である旨・残る blocker の内容・続行/打ち切りの選択肢をユーザーに提示して判断を仰ぐ（続行を選んだ場合の 3 周目以降も同一エージェントへの `SendMessage` 差し戻しで行う）

### 5. plan モード復帰
- plan モード中の場合のみ、修正後の計画で `ExitPlanMode` を呼び出す
- plan モード外で起動された場合は、レビュー結果と反映内容の報告のみで終了

## 注意事項
- @~/.claude/skills/check-plan-compliance/SKILL.md（プロジェクト規約への準拠検証）と併用する場合は **compliance 先 → 本スキル後** の順で実行する（高コストな検証は最終形の計画に 1 回だけ充てるため）。その場合 ExitPlanMode は本スキル側が担い、compliance 側では呼ばない
- 検証モデルは `opus` 指定（2026-07-15 に `fable` から変更）: 実測で検出できた blocker は参照実在性クラスでモデルティアの影響が小さく、fable の単価（opus の 2 倍、2026-07-20 以降はサブスクリプション外の従量課金）に見合わないため。設計妥当性の批評を厚くしたい場合のみ手順 2 の model を `fable` に変更する（上位ティアの批評は完了時の deep-review 側に残っている）
- 検証サブエージェントは読み取り専用で、計画の修正は本スキル（親セッション）が行う
